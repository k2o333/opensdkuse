## P0：必须先修，不然会影响“符合文档/可稳定交付”

### 1. 把 `--json` 从“JSON 打印模式”改成“真正的 structured output 模式”

**问题**
现在 `main.ts` 里，`cliArgs.json` 只会触发 `formatJsonOutput(normalized)`，但 `executePrompt()` 只有在 `opts.structured` 存在时才会传 `format: { type: "json_schema", schema, retryCount }`；而 `main.ts` 构造的 `promptOpts` 目前只塞了 `model`，没有塞任何 schema。README 却把 `--json` 写成了“JSON 结构化输出”。([GitHub][2])

**影响**
用户以为 `--json` 会请求模型按 schema 返回结构化结果，但实际上只是把普通响应包装成 JSON 输出。这和 README、以及你前面的 PRD/TDD 目标都不一致。([GitHub][3])

**建议改法**
在 `main.ts` 里，当 `cliArgs.json === true` 时，必须同时构造 `promptOpts.structured`。MVP 可以先选一种最简单实现：

* 方案 A：内置一个默认 schema，适用于“抽取字段”类任务；
* 方案 B：新增 `--schema-file <path>`；
* 方案 C：如果没有 schema，就禁止 `--json` 并给出清晰报错。

**涉及文件**

* `src/main.ts`
* `src/opencode.ts`
* `src/response.ts`
* `src/cli.ts`
* `README.md`

**验收标准**

* `--json` 时，`executePrompt()` 实际传入 `format.type = "json_schema"`
* 返回结果里能拿到 structured output
* 没有 schema 时行为明确，不是“默默降级成普通文本 JSON 打印”
  ([GitHub][2])

### 2. 明确并修正 SDK 返回结构兼容：`responseStyle`、`structured_output`、错误字段

**问题**
官方文档说明 `createOpencodeClient()` 默认 `responseStyle` 是 `fields`，不是 `data`；结构化输出示例读取的是 `result.data.info.structured_output`；StructuredOutputError 示例读取的是 `result.data.info.error.message` 和 `result.data.info.error.retries`。但你当前实现里：

* attach 时 `createOpencodeClient({ baseUrl })` 没有显式设置 `responseStyle`
* `response.ts` 读取的是 `info.structured`
* `response.ts` 读取错误是 `info.error.data?.message` 和 `info.error.data?.retries`
* 业务层大量依赖 `result.data`。([GitHub][4])

**影响**
如果你切到官方推荐的公开 SDK 调用形态，或者未来升级 SDK，这一层很容易直接失效。尤其是 `responseStyle` 默认值和结构化字段名不一致，会导致正常结果被解析为空。([OpenCode][5])

**建议改法**
`src/opencode.ts` 和 `src/response.ts` 做一次“兼容层重构”：

在 `connectOrStartServer()` attach 路径中，显式设置：

```ts
createOpencodeClient({
  baseUrl,
  responseStyle: "data",
})
```

在 `response.ts` 里同时兼容：

* `info.structured_output`
* `info.structured`

以及错误字段同时兼容：

* `info.error.message`
* `info.error.retries`
* `info.error.data?.message`
* `info.error.data?.retries`

**涉及文件**

* `src/opencode.ts`
* `src/response.ts`
* `test/response.test.ts`
* `test/opencode.test.ts`

**验收标准**

* 使用 `responseStyle: "data"` 时可稳定工作
* 对 `structured_output` / `structured` 都能解析
* 对 `error.message` / `error.data.message` 都能解析
  ([GitHub][4])

### 3. 修复 signal handler 的解绑 bug

**问题**
你注册 signal 时用的是匿名函数：

```ts
process.on("SIGINT", () => onSignal("SIGINT"))
process.on("SIGTERM", () => onSignal("SIGTERM"))
```

但 finally 里解绑时又重新创建了新的匿名函数：

```ts
process.off("SIGINT", () => onSignal("SIGINT"))
process.off("SIGTERM", () => onSignal("SIGTERM"))
```

这不会移除原来的监听器。([GitHub][2])

**影响**
单次 CLI 跑完不一定炸，但在测试反复执行、或未来嵌入更长生命周期进程时，会累积监听器，造成重复 abort、日志重复或内存泄漏。

**建议改法**
改成具名 handler：

```ts
const handleSigint = () => onSignal("SIGINT")
const handleSigterm = () => onSignal("SIGTERM")

process.on("SIGINT", handleSigint)
process.on("SIGTERM", handleSigterm)

...

process.off("SIGINT", handleSigint)
process.off("SIGTERM", handleSigterm)
```

**涉及文件**

* `src/main.ts`
* `test/main.test.ts`

**验收标准**

* 多次调用 `main()` 后监听器数量不增长
* `SIGINT` / `SIGTERM` 只触发一次 abort 路径
  ([GitHub][2])

### 4. `createSession()` 必须真正消费 `agent / model / permissions`

**问题**
`main.ts` 在调用 `createSession()` 时已经传了 `title`、`agent`、`model`，但 `opencode.ts` 中 `createSession()` 只把 `title` 传给 `client.session.create()`，其他字段都被丢了。([GitHub][2])

**影响**
CLI 对外暴露了 `--agent` / `--model` 能力，但 session 创建阶段并没有真正落实。用户看到参数可用，实际行为却不一定生效。

**建议改法**
按你最终要支持的 SDK 版本做两件事之一：

* 如果 session.create 支持这些字段，就透传；
* 如果 session.create 不支持，就删除 `SessionConfig` 里的误导字段，并在 `main.ts`/README 说明模型只作用在 `session.prompt()` 阶段、agent 只做校验或另有入口。

官方公开文档里 `session.create()` 是 `client.session.create({ body: { title: ... } })`，而 prompt 才明确支持 `model`。因此这块最重要的是“行为和文档一致”，不要让 CLI 参数看上去支持，实际上没落地。([GitHub][2])

**涉及文件**

* `src/opencode.ts`
* `src/main.ts`
* `README.md`

**验收标准**

* `--agent` / `--model` 的实际作用位置被明确实现
* 代码、README、help 文案三者一致
  ([GitHub][2])

### 5. 决定“公共 SDK API”还是“v2 内部路径”，不要两头悬空

**问题**
仓库当前用的是 `@opencode-ai/sdk/v2` 和 `@opencode-ai/sdk/v2/client`，并采用 `sessionID`、顶层 `title`、顶层 `noReply` 这类调用形态；但官方文档当前展示的是从 `@opencode-ai/sdk` 直接导入，调用形态是 `client.session.create({ body })`、`client.session.prompt({ path, body })`。([GitHub][4])

**影响**
现在代码可能“在你本机能跑”，但外部读者或未来你自己升级 SDK 时，很难判断你究竟绑定的是哪套稳定契约。维护成本会很高。

**建议改法**
二选一：

* **方案 A：迁到官方公开 API 形态**
  用 `@opencode-ai/sdk` 公共入口，按文档的 `path/body` 调用方式重写适配层。

* **方案 B：继续使用 v2 内部路径**
  那就必须：

  1. 把 `package.json` 版本锁死；
  2. 在 README 顶部明确“本项目绑定某个 v2 API 版本”；
  3. 在 `src/opencode.ts` 顶部写注释说明与官方公开文档的差异。

对 MVP 来说，我更建议 A。([GitHub][4])

### 6. 锁定 SDK 版本，不要使用 `"latest"`

**问题**

**涉及文件**

* `package.json`

**验收标准**

* `npm install` 后版本可重复
* README 中有“当前验证 SDK 版本”说明
  ([GitHub][6])

### 7. 拆分或重命名 `startupTimeoutMs`

**问题**
`config.ts` 里的字段叫 `startupTimeoutMs`，默认 30000；但在 `main.ts` 里它被用作整次执行超时，而在 `opencode.ts` 里又被拿去做 spawn server 的 `timeout`。也就是说，一个字段同时承担了“server 启动超时”和“任务执行总超时”两种含义。([GitHub][7])

**影响**
语义混乱。你以后很难单独调“server 启动慢一点，但任务总时长更长”这种配置。

**建议改法**

* 方案 A：拆成

  * `serverStartupTimeoutMs`
  * `executionTimeoutMs`
* 方案 B：如果暂时只保留一个，就把名字改成 `executionTimeoutMs`，并在 spawn 时单独写一个固定或派生值

**涉及文件**

* `src/config.ts`
* `src/main.ts`
* `src/opencode.ts`
* `README.md`
* `test/config.test.ts`

**验收标准**

* 配置项语义单一
* help/README 与代码一致
  ([GitHub][7])

## P2：质量提升项，不修也能跑，但修了会更稳

### 11. `response.ts` 做成“版本兼容适配层”，别只适配当前猜测结构

**问题**
现在 `response.ts` 已经有归一化层，这很好；但它仍然过度假设某一套返回结构，尤其是 `info.structured`、`info.error.data`。([GitHub][8])

**建议改法**
把 `normalizeSdkResponse()` 改成容错式：

* 先识别是否存在 `result.data`
* 再识别 `fields` 风格
* structured 兼容多个字段名
* error 兼容多个层级
* 把解析逻辑写成小函数，便于单测

**价值**
后续升级 SDK 时，你只需要改这一层。
([OpenCode][5])
